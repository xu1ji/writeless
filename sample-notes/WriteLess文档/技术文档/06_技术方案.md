## 为什么选择这套技术栈

### 1. Tauri vs Electron

| 对比 | Tauri | Electron |
|------|-------|----------|
| 包体积 | ~10MB | ~150MB |
| 内存占用 | ~100MB | ~300MB |
| 启动速度 | 快 | 慢 |
| 原生能力 | Rust，强大 | Node.js |
| 生态 | 较新 | 成熟 |
| 学习曲线 | Rust 有门槛 | JS 友好 |

**选择 Tauri 的理由**：
- 轻量：用户下载快，运行省资源
- 安全：Rust 内存安全
- 性能：Rust 处理文件监听、AI 计算更高效

### 2. ProseMirror vs CodeMirror

| 对比 | ProseMirror | CodeMirror |
|------|-------------|------------|
| 定位 | 富文本编辑框架 | 代码编辑器 |
| 结构化 | ✅ 原生支持 | ⚠️ 需要扩展 |
| Markdown | 需要实现 | ✅ 内置 |
| 学习曲线 | 陡峭 | 平缓 |
| 灵活性 | 极高 | 中等 |

**选择 ProseMirror 的理由**：
- 结构化内容：更适合有元数据的笔记
- 灵活性：可以自定义块级元素（关联面板、AI 建议等）
- 长期维护：已被多个大产品验证（Notion、Confluence）

### 3. SQLite vs 其他

| 对比 | SQLite | IndexedDB | 文件系统 |
|------|--------|-----------|----------|
| 查询能力 | ✅ SQL | ⚠️ 有限 | ❌ 无 |
| 性能 | ✅ 快 | ⚠️ 一般 | ⚠️ IO 限制 |
| 可移植性 | ✅ 单文件 | ❌ 浏览器 | ✅ 文件 |
| 并发 | ⚠️ 有限 | ✅ 好 | ❌ 锁 |

**选择 SQLite 的理由**：
- 向量搜索：sqlite-vec 扩展支持向量索引
- 单文件：方便备份和迁移
- 成熟：稳定可靠

## AI 模型选择

### 本地模型

| 用途 | 模型 | 大小 | 硬件要求 |
|------|------|------|----------|
| Embedding | all-MiniLM-L6-v2 | 80MB | 低 |
| 简单生成 | Phi-3 Mini | 2.4GB | 中 |
| 复杂生成 | LLaMA 3 8B | 4.7GB | 高 |

### 云端模型（可选）

| 用途 | 模型 | 特点 |
|------|------|------|
| 复杂问答 | Claude 3.5 Sonnet | 理解能力强 |
| 快速生成 | Claude 3.5 Haiku | 速度快、便宜 |
| 备选 | GPT-4o | 生态好 |

### 混合策略

```typescript
const aiConfig = {
  embedding: {
    provider: 'local',
    model: 'all-MiniLM-L6-v2',
  },

  summary: {
    provider: 'local',  // 优先本地
    model: 'phi-3-mini',
    fallback: {         // 失败时云端
      provider: 'anthropic',
      model: 'claude-3-haiku',
    },
  },

  complexQA: {
    provider: 'anthropic',
    model: 'claude-3.5-sonnet',
  },
};
```

## 关键技术挑战

### 1. 大文件性能

**问题**：单篇笔记可能很长，ProseMirror 需要优化

**方案**：
- 虚拟滚动
- 增量渲染
- 延迟加载大型块

### 2. 向量索引性能

**问题**：笔记数量增长后，向量搜索变慢

**方案**：
- 使用 HNSW 索引（LanceDB）
- 增量更新，避免全量重建
- 分片存储

### 3. AI 响应延迟

**问题**：AI 生成需要时间，用户体验差

**方案**：
- 后台预生成摘要
- 流式返回
- 缓存常见查询

### 4. 跨平台一致性

**问题**：Mac/Windows/Linux 文件系统差异

**方案**：
- 使用 Tauri 的跨平台 API
- 统一路径处理
- 充分测试
