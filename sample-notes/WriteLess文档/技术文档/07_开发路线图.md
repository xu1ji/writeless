# 07. 开发路线图

> 分阶段验证产品内核的开发计划

---

## 核心原则

**开发顺序 = 验证优先级**

不是"先做基础再做高级"，而是"先验证最重要假设"。

```
假设验证优先级：
  K4: 分层索引 ─┐
  K3: 自动关联 ─┼─▶ 最重要，最先验证
  K2: 常驻 AI  ─┤
  K1: 全局理解 ─┘
```

## Phase 1: 基础设施 (2-3 周)

> 目标：能跑起来的最小骨架

### 功能清单

| 功能 | 说明 | 必要性 |
|------|------|--------|
| **编辑器** | 能写 Markdown | 必须有 |
| **本地存储** | 能保存文件 | 必须有 |
| **文件管理** | 能新建/删除 | 必须有 |

### 成功标准

- [ ] 能创建、编辑、保存 Markdown 笔记
- [ ] 能在文件树中浏览笔记
- [ ] 数据存储在本地文件夹

### 不做

- ❌ 不做高级编辑功能（表格、图片）
- ❌ 不做主题系统
- ❌ 不做移动端
- ❌ 不做协作

---

## Phase 2: 验证 K1 - AI 全局理解 (2-3 周)

> 目标：验证"AI 能理解语义，不只是关键词"

### 假设

**如果实现语义搜索，用户满意度 > 70%，则假设成立。**

### 功能清单

| 功能 | 说明 |
|------|------|
| **本地 Embedding** | all-MiniLM-L6-v2 |
| **向量存储** | sqlite-vec |
| **语义搜索 UI** | 搜索框 + 结果列表 |
| **AI 摘要** | 为每篇笔记生成一句话摘要 |

### 验证方式

1. 导入 50+ 篇笔记
2. 用自然语言搜索（不是关键词）
3. 记录搜索满意度

### 成功标准

- [ ] 语义搜索使用率 > 60%
- [ ] 搜索满意度 > 70%（用户访谈）
- [ ] AI 摘要有用率 > 50%

### 决策

- ✅ 成功 → 继续 Phase 3
- ❌ 失败 → 调整方向或放弃

---

## Phase 3: 验证 K2 - 常驻 AI 大脑 (1-2 周)

> 目标：验证"AI 后台索引，主动建议"

### 假设

**如果 AI 主动给出建议，点击率 > 20%，则假设成立。**

### 功能清单

| 功能 | 说明 |
|------|------|
| **后台索引** | 文件保存时自动触发 |
| **实时建议** | 编辑时 AI 建议（简单浮层） |
| **自动标签** | AI 推荐标签 |

### 验证方式

1. 记录 AI 建议展示次数
2. 记录用户点击次数
3. 计算点击率

### 成功标准

- [ ] AI 建议点击率 > 20%
- [ ] 索引覆盖率 100%

### 决策

- ✅ 成功 → 继续 Phase 4
- ❌ 失败 → 调整建议算法

---

## Phase 4: 验证 K3 - 自动关联 (1-2 周)

> 目标：验证"AI 能发现笔记之间的关联"

### 假设

**如果 AI 发现关联，用户会有"惊喜感"反馈，则假设成立。**

### 功能清单

| 功能 | 说明 |
|------|------|
| **关联推荐** | 基于向量相似度 |
| **反向链接面板** | 显示哪些笔记链接到当前笔记 |
| **关联提示** | "这篇笔记和 X 相关" |

### 验证方式

1. 记录关联推荐点击率
2. 用户访谈：有没有"惊喜感"？
3. NPS 调研

### 成功标准

- [ ] 关联推荐点击率 > 20%
- [ ] 用户反馈有"惊喜感"
- [ ] NPS > 40

### 决策

- ✅ 成功 → 继续 Phase 5
- ❌ 失败 → 调整关联算法

---

## Phase 5: 验证 K4 - 分层索引 (2-3 周)

> 目标：验证"分层索引比 RAG 更精准"

### 假设

**如果分层索引的检索准确率比 RAG 提升 30%+，则假设成立。**

### 功能清单

| 功能 | 说明 |
|------|------|
| **L1-L3 生成** | AI 生成摘要/大纲/全文 |
| **分层存储** | 存储三层内容 |
| **层级检索** | 从 L1 开始逐层检索 |
| **对比测试** | A/B 测试分层 vs RAG |

### 验证方式

1. 准备 100+ 篇笔记
2. 同样查询，对比分层 vs RAG 结果
3. 用户判断哪个更准确

### 成功标准

- [ ] 分层索引准确率 > RAG +30%
- [ ] 检索速度可接受（< 500ms）

### 决策

- ✅ 成功 → 核心假设全部验证，进入 Phase 6
- ❌ 失败 → 评估是否值得继续

## 风险与应对

| 风险 | 应对 |
|------|------|
| AI 模型本地运行慢 | 混合方案，云端备用 |
| 向量索引内存大 | 量化压缩 |
| 分层索引不比 RAG 好 | 调整算法或放弃这个假设 |
| 用户不点 AI 建议 | 改进建议质量或放弃 |

---

## 关键原则

1. **验证优先**：先验证假设，再完善功能
2. **低成本验证**：用最小功能验证最大假设
3. **快速失败**：假设不成立就快速调整
4. **不提前优化**：验证成功后再优化体验
